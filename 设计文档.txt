一、项目概述
================================================================================

本项目是一款基于大语言模型的狼人杀游戏。用户自带 API Key，纯前端运行，
无需后端服务器。支持两种模式：

  【模式A：人机混合】玩家 + AI 角色一起游玩
  【模式B：赛博斗蛐蛐】纯 AI 互相博弈，玩家观战

核心卖点：
  - 用户自行接入任意 AI（OpenAI / DeepSeek / Gemini / Claude / 自定义）
  - API Key 仅存储在用户浏览器 localStorage，不经过任何服务器
  - 每个 AI 座位可配置不同模型（类似 cc-switch 切换体验）
  - 赛博斗蛐蛐模式：不同大模型之间对战，玩家观看博弈过程
  - 赛博斗蛐蛐下注系统：角色分配后用户可下注猜测获胜AI，结算页展示结果
  - 免费模型即可胜任（GPT-4o-mini、Gemini Flash 有免费额度）

四、核心类型定义
================================================================================

// 游戏阶段
type GamePhase =
  | 'idle'              # 未开始
  | 'role_assign'       # 分配角色
  | 'betting'           # 下注环节（赛博斗蛐蛐模式专属）
  | 'night_start'       # 夜晚序幕
  | 'werewolf_turn'     # 狼人行动
  | 'seer_turn'         # 预言家行动
  | 'witch_turn'        # 女巫行动
  | 'night_end'         # 结算夜晚死亡
  | 'day_start'         # 公布死亡
  | 'day_speech'        # 白天发言
  | 'vote_start'        # 投票
  | 'vote_result'       # 投票结算
  | 'check_win'         # 胜负检查
  | 'game_over'         # 游戏结束

// 角色
type Role = 'werewolf' | 'villager' | 'seer' | 'witch' | 'hunter'

// 游戏模式
type GameMode = 'human-ai' | 'ai-only'

// 玩家
interface Player {
  id: number
  name: string
  role: Role
  isAlive: boolean
  isAI: boolean
  modelConfig?: AIConfig       // 该位置使用的 AI 配置
  personality?: string          // 模型个性描述（赛博模式）
}

// AI 配置（用户输入）
interface AIConfig {
  provider: 'openai' | 'deepseek' | 'gemini' | 'anthropic' | 'custom' | 'mock' | 'built-in'
  model: string
  apiKey: string
  baseURL?: string
}

// 游戏状态
interface GameState {
  phase: GamePhase
  mode: GameMode
  round: number
  players: Player[]
  nightResult: {
    killedId: number | null     // 狼人目标
    savedId: number | null      // 女巫救人
    poisonedId: number | null   // 女巫毒人
    seerCheckedId: number | null
    seerCheckedResult: boolean | null
  }
  messages: GameMessage[]
  votes: Record<number, number> // voterId -> targetId
  winner: 'villager' | 'werewolf' | null
  waitingForInput: boolean      // 是否等待真人玩家输入
  betPlayerId: number | null    // 用户下注的AI玩家ID（赛博模式）
  betResult: boolean | null     // 下注结果：true=猜对, false=猜错
}

// 发言消息
interface GameMessage {
  playerId: number
  playerName: string
  content: string
  round: number
  phase: GamePhase
  modelProvider?: string        // 赛博模式标注用了哪个模型
  isThinking?: boolean          // 标记为AI思考过程消息
}

// 对局历史
interface GameHistory {
  id: string
  date: string
  mode: GameMode
  playerCount: number
  winner: 'villager' | 'werewolf' | null
  rounds: number
  models: string[]
  players: Player[]
  messages: GameMessage[]
  betPlayerId?: number | null   // 用户下注的玩家ID
  betResult?: boolean | null    // 下注结果
}

================================================================================
五、游戏状态机（GameEngine）
================================================================================

核心驱动方法：nextPhase()，根据当前阶段自动跳转。

流程：
  idle → role_assign
       → [赛博斗蛐蛐模式] betting（用户下注猜测获胜AI）
       → night_start → werewolf_turn → seer_turn → witch_turn
       → night_end → 结算死亡
       → check_win → (未结束) day_start
       → day_speech → 轮流发言
       → vote_start → vote_result → 放逐 + 遗言
       → check_win → (未结束) 回到 night_start

关键设计：
  1. 顺序严格：每个 phase 只做一件事，完成后才进入下一个
  2. 异步等待：AI 调用和玩家输入都是 async/await，不阻塞 UI
  3. 信息隔离：夜晚结果存在 nightResult，结算前不让任何人看到
  4. 跳过逻辑：角色已死则跳过该角色的回合
  5. 真人解耦：用 Promise 等待玩家输入，UI 和引擎完全分离
  6. 两处判断胜负：夜晚结算后 + 投票结算后
  7. 下注环节：赛博斗蛐蛐模式在 role_assign 后插入 betting 阶段
     → 用户选择猜测获胜的AI → 游戏结束时自动结算下注结果

胜负条件：
  - 狼人数量 === 0 → 好人阵营胜利
  - 狼人数量 >= 好人数量 → 狼人阵营胜利

================================================================================
六、Prompt 模板设计
================================================================================

核心策略：四层分层结构
  第一层：系统人设（你是谁，你的目标）
  第二层：游戏规则（狼人杀规则说明）
  第三层：当前状态（动态注入：第几天、谁活着、历史发言）
  第四层：行为指令（本轮要你做什么）

[基础规则层 - 所有角色共享]
  基本规则、发言要求、行为约束

[村民 Prompt]
  策略：通过观察和推理找出狼人
  分析发言一致性，注意谁在帮谁说话

[狼人 Prompt - 最复杂]
  信息：包含队友名单
  夜晚：选择击杀目标（优先预言家 > 女巫 > 犀利村民）
  白天：伪装好人，不能暴露队友，适度带节奏
  投票：跟票好人，保护队友

[预言家 Prompt]
  信息：所有已查验结果
  策略：权衡是否公开身份（公开→被信任但被盯上；隐藏→安全但无法引导）
  查验优先级：发言异常的人

[女巫 Prompt]
  技能：解药（1次）、毒药（1次），使用状态实时告知
  解药策略：第一晚自救？预言家被杀时救？
  毒药策略：确认狼人再用

[猎人 Prompt]
  技能：死亡时开枪带走一人（被毒死不能开枪）

输出格式约束：
  - 行动类（夜晚技能、投票）：强制 JSON 格式
  - 发言类（白天讨论）：纯文本，80 字以内
  - 加容错解析：尝试 JSON.parse，失败则提取 {.*} ，再失败用默认值

状态注入函数 buildGameState()：
  动态替换：当前轮次、存活列表、死亡列表、历史发言（最多20条）、投票记录

赛博斗蛐蛐特殊设计：
  不同模型给不同"人设标签"：
    GPT-4o    → "逻辑严密，喜欢引用前面发言"
    Claude     → "谨慎保守，关键时刻一针见血"
    DeepSeek  → "直接果断，有时过于自信"
    Gemini     → "活跃话多，但偶尔偏题"
  每个发言气泡标注来源于哪个模型

================================================================================
七、AI 调用层（AIClient）
================================================================================

统一接口，兼容所有主流模型：

  async function callAI(config: AIConfig, messages: Message[]): Promise<string>

默认 baseURL 映射：
  OpenAI   → https://api.openai.com/v1
  DeepSeek → https://api.deepseek.com/v1
  Gemini   → https://generativelanguage.googleapis.com/v1beta/openai
  Anthropic→ https://api.anthropic.com/v1
  Custom   → 用户自定义 URL

支持流式输出（stream: true），发言像打字机一样逐字出现。

配置存储：
  所有 API Key 和模型选择存入 localStorage
  键名：werewolf-ai-config
  离开页面时可用 beforeunload 提示保存

十、成本分析
================================================================================

单局估算（8人局，平均4轮）：
  每轮发言：8人 × 200 token = 1,600 token
  夜晚技能：3-5 次调用 × 300 token = 1,500 token
  投票+遗言：约 1,000 token
  总计约 8,000 ~ 10,000 token/局

使用 GPT-4o-mini：约 $0.001/局
使用 Gemini Flash：约 $0.000/局（免费额度内）
使用 DeepSeek：约 ¥0.008/局

运营方成本：零（纯前端，用户自带 Key）

================================================================================

首选：Vercel
  - 免费静态托管

  - 全球 CDN 加速

================================================================================
十三、注意事项与风险
================================================================================

1. API Key 安全
   - 纯前端方案，Key 存在用户浏览器
   - 不上传任何服务器
   - 明确告知用户风险

2. AI 输出不可控
   - Prompt 必须严格约束输出格式
   - 行动指令强制 JSON
   - 发言指令限制字数
   - 加多层容错解析

3. 延迟体验
   - AI 响应 1-3 秒
   - 使用流式输出减少等待感
   - 赛博模式可自动推进，无需用户等待

4. 上下文溢出
   - 历史发言只保留最近 20 条
   - 定期总结压缩

5. 微信小程序限制
   - 暂不做小程序
   - 如后续需要，需自建中转服务器

6. 跨域问题
   - 部分 AI API 可能不支持浏览器直接调用（CORS）
   - 可能需要配置 Vite 代理或提示用户使用中转
